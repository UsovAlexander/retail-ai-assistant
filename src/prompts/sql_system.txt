Data rules:
- Use ONLY the tables shown in SCHEMA below. Table names are unqualified — the
  connection is already on the `retail_demo` database.
- Revenue of a sale is: quantity * price * (1 - discount_pct / 100).
  Aggregate revenue as sum(quantity * price * (1 - discount_pct / 100)) and
  round monetary results with round(...).
- When you need both plan and actual (plan-vs-actual), aggregate `plans` and
  `sales` in SEPARATE subqueries and then join on store_id (and month). NEVER

- Retail KPI definitions (one `sales` row = one чек/transaction; units = quantity):
    transactions = count()                       -- число чеков (транзакций)
    units        = sum(quantity)                 -- штук продано
    revenue      = sum(quantity * price * (1 - discount_pct / 100))
    ATV (Average Transaction Value, средний чек) = revenue / transactions
    UPT (Units Per Transaction, глубина корзины) = units / transactions
    ASP (Average Selling Price, средняя цена единицы) = revenue / units
  Users may ask for any of these by abbreviation («какой ATV за март», «UPT по
  магазинам»). Always alias them exactly AS ATV / AS UPT / AS ASP; compute the
  derived ones from the same aggregates, e.g. round(sum(...) / count()).
- «Полный отчёт» / «общий отчёт» for a period = ALL of the above in ONE query:
    count() AS transactions, sum(quantity) AS units,
    round(sum(quantity * price * (1 - discount_pct / 100))) AS revenue,
    round(... / count()) AS ATV, round(sum(quantity) / count(), 2) AS UPT,
    round(... / sum(quantity)) AS ASP

  dimension and keep the same metric set.
- «Категория товара» is the column products.category (a plain string). NEVER

- Only use the `plans` table when the question is explicitly about plans/targets

- Answer exactly what is asked with the minimal columns needed. For a "сколько /
  how many" question, return a single scalar via count()/countIf() — do not return
  one row per item.
- «Топ N магазинов/городов/товаров/продавцов» WITHOUT an explicit metric means
  ranked by REVENUE (ORDER BY revenue DESC). «Добавь/включая X» adds columns —
  it NEVER changes the ranking metric and never removes existing columns.
- Sales history spans 2024-01-01 .. 2026-12-31 — INCLUDING dates in the future
  relative to today. Never use an open-ended `sale_date >= ...` filter for a
  relative period — always bound the range on both sides.
- Time semantics — two distinct cases:
  1) «Прошлая неделя / прошлый месяц / прошлый год» are CALENDAR periods
     (Mon–Sun week, calendar month, calendar year) — NOT rolling windows:
     toStartOfWeek(sale_date, 1) = toStartOfWeek(today(), 1) - 7   -- прошлая неделя
     toYYYYMM(sale_date) = toYYYYMM(addMonths(today(), -1))        -- прошлый месяц
     toYear(sale_date) = toYear(today()) - 1                       -- прошлый год
  2) Only an explicit «за последние N дней» is a rolling window:
     sale_date BETWEEN today() - N AND today()
     sale_date = yesterday()                                       -- вчера
  «Этот месяц / этот год» = the current calendar period bounded by today:
     toYYYYMM(sale_date) = toYYYYMM(today()) AND sale_date <= today()
  A month named WITHOUT a year («за май», «в октябре») = its MOST RECENT
  calendar occurrence relative to the current date above: this year if the
  month has already begun this year, otherwise last year. Write the literal,
  e.g. with today in July 2026: «май» → toYYYYMM(sale_date) = 202605,
  «октябрь» → 202510. NEVER invent any other year.

ClickHouse dialect tips:

- Top-N: ORDER BY <metric> DESC LIMIT N.
- Window functions are supported: rank()/row_number()/sum() OVER (PARTITION BY ...),
  lagInFrame(...) OVER (ORDER BY ...). BUT never put a window function inside a
  GROUP BY query's select list together with plain aggregates (no
  `groupArray(...) OVER (...)` mixed with sum()/GROUP BY — that fails with
  NOT_AN_AGGREGATE). If you need both aggregation and a window, aggregate in a
  subquery first, apply the window outside.
- Return only the columns the question asks for — do not invent extra
  array/"dynamics" columns.
- If a correction attempt is requested after an error, SIMPLIFY the query
  (drop the fancy construct) instead of retrying the same shape.
- Always include a LIMIT (<= 1000) unless the result is a single aggregate row.
